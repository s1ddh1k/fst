# 알고리즘 개발 가드레일

이 문서는 `fst`의 알고리즘 개발 방향을 고정하기 위한 작업 합의문이다.

## 목표

- 멀티 상품을 동시에 평가하는 cross-sectional 구조를 기본으로 한다.
- 여러 지표를 통합한 점수화 또는 포트폴리오 의사결정 구조를 기본으로 한다.
- 특정 단일 자산에 의존하지 않는 시장 해석 구조를 만든다.
- 연구, 추천 저장, paper-trader가 같은 구조적 가정을 공유하게 만든다.

## 금지사항

- `BTC`를 특권적 기준 자산으로 두지 않는다.
- 단일 종목 기준 해석을 전체 시장 해석으로 포장하지 않는다.
- 실행 검증 편의를 이유로 아키텍처 방향을 임의로 바꾸지 않는다.
- 임시 smoke 설정을 명시 없이 기본 구조에 섞어 넣지 않는다.

## 기본 원칙

- 벤치마크는 단일 종목보다 `유니버스 합성 지수` 또는 `유니버스 내부 상대 기준`을 우선한다.
- 상대강도는 `vs 특정 코인`보다 `vs 유니버스 composite` 또는 `vs 그룹 평균`을 우선한다.
- regime은 단일 자산 추세보다 `breadth`, `dispersion`, `liquidity`, `composite trend` 축으로 해석한다.
- 전략 검증을 위해 임시 설정이 필요하면 `temporary`, `smoke-only`, `not-for-architecture` 중 하나로 명시한다.

## 변경 통제 규칙

다음 항목은 방향 변경으로 간주하며 사전 확인 없이 넣지 않는다.

- 특정 자산을 benchmark로 고정하는 변경
- 단일 종목 fallback을 기본 경로에 넣는 변경
- 검증 편의를 이유로 feature 정의나 추천 구조를 바꾸는 변경
- 알고리즘 목표와 무관한 임시 규칙을 live 경로에 넣는 변경

## 작업 시작 체크리스트

각 구현 작업은 시작 전에 아래를 짧게 고정한다.

- 이번 작업의 목표
- 이번 작업의 비목표
- 이번 작업에서 유지할 금지사항
- 필요한 예외가 있다면 그 이유와 범위

## 리뷰 체크리스트

코드나 문서를 추가할 때 아래 질문에 `예`가 나와야 한다.

- 이 변경은 멀티 상품 구조를 강화하는가?
- 이 변경은 멀티 지표 통합 해석을 강화하는가?
- 이 변경은 특정 코인 중심 편향을 줄이는가?
- 이 변경은 연구 경로와 paper 경로의 구조를 더 일치시키는가?

아래 질문에 `아니오`가 나와야 한다.

- 이 변경이 검증 편의 때문에 방향을 틀고 있는가?
- 이 변경이 임시 구현을 기본 구조처럼 숨기고 있는가?
- 이 변경이 단일 종목 관점을 시장 전체 관점처럼 취급하는가?

## 예외 처리

예외가 꼭 필요하면 아래를 같이 남긴다.

- 왜 예외가 필요한지
- 어디까지가 임시 범위인지
- 제거 또는 교체 시점이 언제인지

예외는 기본 설계로 승격하지 않는다.
