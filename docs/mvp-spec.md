# 업비트 자동매매 앱 MVP 기능 명세서

## 1. 문서 목적

이 문서는 프로젝트 계획 문서를 바탕으로 1차 MVP 범위를 확정하기 위한 기능 명세서다.

목표는 다음 세 가지다.

- 지금 바로 구현할 기능을 명확히 정한다.
- 구현하지 않을 기능을 의도적으로 제외한다.
- 이후 저장소 초기화와 개발 작업의 기준 문서로 사용한다.

## 2. MVP 목표

MVP의 목표는 `실거래 자동매매 앱 완성`이 아니다.

MVP의 실제 목표는 다음과 같다.

- 업비트 과거 데이터를 안정적으로 수집하고 저장한다.
- 최소 3개 이상의 전략을 동일한 조건으로 백테스트한다.
- 최근 1년 미학습 구간에서 전략 성능을 비교한다.
- 선택한 전략을 실시간 모의투자로 실행한다.
- Electron 앱에서 전략 결과와 실시간 모의투자 상태를 확인한다.

즉, MVP는 `데이터 -> 연구 -> 검증 -> 모의투자 -> 시각화`까지를 완성하는 단계다.

## 3. MVP 범위

### 3.1 포함 기능

#### A. 데이터 수집

- 업비트 REST API 기반 과거 캔들 수집
- 대상 마켓:
  - KRW-BTC
  - KRW-ETH
  - KRW-XRP
  - KRW-SOL
  - KRW-DOGE
- 대상 주기:
  - 1분봉
  - 5분봉
  - 1시간봉
  - 일봉
- 장기 적재 실행 스크립트
- 누락 구간 재수집 기능
- 수집 로그 저장

#### B. 데이터 저장

- 원본 캔들 데이터 저장
- 정규화된 분석용 데이터셋 생성
- 전략 실행 결과 저장
- 모의투자 주문/체결/포지션 로그 저장

#### C. 백테스트 엔진

- 공통 전략 인터페이스
- 캔들 기반 시그널 생성
- 시장가 매수/매도 가정
- 수수료 반영
- 슬리피지 반영
- 포지션 및 잔고 추적
- 백테스트 결과 리포트 생성

#### D. 전략 라이브러리

- 이동평균 교차 전략
- 변동성 돌파 전략
- RSI 평균회귀 전략

필요시 다음 전략 중 1~2개를 추가한다.

- Bollinger Band 평균회귀
- 추세 필터 + 손절/익절 전략

#### E. 검증

- 전체 데이터 중 최근 1년 홀드아웃 검증
- 전략별 성능 비교표 생성
- 주요 지표 계산:
  - 총 수익률
  - 연환산 수익률
  - 최대 낙폭
  - Sharpe 또는 Sortino
  - 거래 횟수
  - 승률
  - Profit Factor

#### F. 실시간 모의투자

- 업비트 WebSocket 시세 수신
- 전략 신호에 따른 가상 주문 생성
- 가상 체결 처리
- 포지션, 잔고, 손익 추적
- 수수료/슬리피지 반영
- 전략 시작/중지 기능

#### G. Electron 앱

- 전략 목록 화면
- 전략별 백테스트 성능 조회 화면
- 전략 선택 후 모의투자 시작/중지 화면
- 실시간 포지션/손익/로그 화면
- 시스템 연결 상태 표시

## 4. MVP 제외 범위

다음 항목은 MVP에서 제외한다.

- 실제 업비트 주문 실행
- 다중 거래소 지원
- 모바일 앱
- 호가 기반 초단타 전략
- 체결 데이터 기반 고빈도 전략
- 딥러닝 기반 예측 모델
- 자동 파라미터 탐색 대규모 분산 처리
- 포트폴리오 최적화 엔진
- 사용자 다계정 지원
- 클라우드 멀티유저 서비스화

이 범위를 초기에 제외해야 프로젝트가 비대해지지 않는다.

## 5. 사용자 시나리오

### 시나리오 1. 데이터 적재

1. 운영자는 수집 스크립트를 실행한다.
2. 지정한 마켓과 캔들 주기의 과거 데이터가 저장된다.
3. 누락 구간이 있으면 재수집이 가능하다.

### 시나리오 2. 전략 비교

1. 운영자는 백테스트를 실행한다.
2. 여러 전략이 동일 데이터와 동일 비용 조건에서 평가된다.
3. 최근 1년 검증 결과를 확인한다.
4. 성능이 상대적으로 안정적인 전략을 선택한다.

### 시나리오 3. 실시간 모의투자

1. 운영자는 Electron 앱에서 전략을 선택한다.
2. 모의투자를 시작한다.
3. 실시간 가격 수신과 함께 가상 주문과 포지션이 업데이트된다.
4. 손익과 로그를 보며 전략 상태를 확인한다.
5. 필요 시 즉시 중지한다.

## 6. 기능 요구사항

### 6.1 데이터 수집기

#### 필수 요구사항

- 지정 마켓과 주기에 대해 과거 캔들을 순차 수집할 수 있어야 한다.
- 마지막 저장 시점을 기준으로 이어받기 수집이 가능해야 한다.
- API 실패 시 재시도 정책이 있어야 한다.
- 수집 성공/실패 로그를 남겨야 한다.
- 수집 결과는 중복 없이 저장되어야 한다.

#### 수용 기준

- 신규 실행 시 과거 데이터 전체 적재가 가능하다.
- 재실행 시 이미 수집된 구간은 중복 삽입되지 않는다.
- 장애 후 재실행 시 누락 구간을 복구할 수 있다.

### 6.2 백테스트 엔진

#### 필수 요구사항

- 전략 입력은 시계열 캔들 데이터다.
- 전략 출력은 매수/매도/대기 신호다.
- 거래 비용과 슬리피지가 적용되어야 한다.
- 손익 곡선, 거래 로그, 성능 지표를 계산해야 한다.
- 학습 구간과 검증 구간을 분리해 결과를 생성해야 한다.

#### 수용 기준

- 동일 입력과 파라미터로 재실행 시 동일 결과가 나온다.
- 전략별 비교 결과를 표 형태로 저장할 수 있다.
- 홀드아웃 1년 결과가 별도로 조회 가능해야 한다.

### 6.3 전략 라이브러리

#### 필수 요구사항

- 최소 3개 전략이 공통 인터페이스로 동작해야 한다.
- 각 전략은 파라미터를 외부에서 주입받을 수 있어야 한다.
- 전략별 결과를 독립적으로 저장할 수 있어야 한다.

#### 수용 기준

- 전략을 추가해도 백테스트 엔진 수정이 최소화되어야 한다.
- 전략 이름과 파라미터, 결과가 함께 기록되어야 한다.

### 6.4 실시간 모의투자 엔진

#### 필수 요구사항

- 실시간 시세 수신이 가능해야 한다.
- 선택 전략에 따라 가상 주문이 발생해야 한다.
- 가상 체결 후 포지션/잔고/손익이 반영되어야 한다.
- 엔진 시작/중지 기능이 있어야 한다.
- 로그와 오류를 저장해야 한다.

#### 수용 기준

- 앱에서 시작한 모의투자가 백엔드 엔진과 연결되어 동작한다.
- 실시간 상태가 화면에 반영된다.
- 엔진 중지 후 상태가 정리된다.

### 6.5 Electron 앱

#### 필수 요구사항

- 전략 목록을 조회할 수 있어야 한다.
- 전략별 백테스트 성능을 확인할 수 있어야 한다.
- 모의투자를 시작/중지할 수 있어야 한다.
- 현재 포지션, 평가손익, 최근 로그를 볼 수 있어야 한다.
- 연결 상태와 오류 상태를 표시해야 한다.

#### 수용 기준

- 비개발자도 앱에서 전략 선택과 모의투자 실행이 가능해야 한다.
- 필수 정보가 새로고침 없이 갱신되어야 한다.

## 7. 비기능 요구사항

### 안정성

- 실시간 엔진 장애 시 앱이 같이 종료되지 않아야 한다.
- WebSocket 연결 끊김 시 자동 재연결 또는 안전 정지 동작이 필요하다.

### 보안

- API Secret은 렌더러 프로세스에 저장하지 않는다.
- 민감 정보는 안전한 저장 방식 또는 로컬 암호화 정책을 따른다.

### 성능

- 모의투자 상태 갱신은 사용자가 체감상 실시간으로 확인 가능해야 한다.
- 백테스트는 최소 수년치 캔들에 대해 반복 실행 가능한 수준이어야 한다.

### 재현성

- 전략 결과는 파라미터, 데이터 버전, 실행 시각과 함께 기록되어야 한다.

## 8. 시스템 구성 제안

MVP는 다음 구조를 기준으로 한다.

- `services/data-collector`
  - 업비트 과거 데이터 수집
- `research/backtester`
  - 백테스트 엔진
- `research/strategies`
  - 전략 구현
- `services/paper-trader`
  - 실시간 모의투자 엔진
- `apps/desktop`
  - Electron 앱
- `packages/shared`
  - 공통 타입, 설정, 유틸리티

## 9. 데이터 모델 초안

MVP에서 필요한 핵심 엔티티는 아래와 같다.

- `markets`
  - 마켓 코드, 상태, 표시명
- `candles`
  - 마켓, 주기, 시각, 시가, 고가, 저가, 종가, 거래량
- `backtest_runs`
  - 전략명, 파라미터, 대상 구간, 실행 시각
- `backtest_metrics`
  - 수익률, MDD, Sharpe, 승률, 거래 횟수
- `paper_sessions`
  - 실행 전략, 시작 시각, 종료 시각, 상태
- `paper_orders`
  - 주문 타입, 가격, 수량, 상태, 생성 시각
- `paper_positions`
  - 보유 수량, 평균 단가, 평가 손익
- `system_logs`
  - 레벨, 이벤트 타입, 메시지, 생성 시각

## 10. 개발 우선순위

MVP는 다음 순서로 구현한다.

1. 저장소 초기 구조 생성
2. 데이터 수집기 구현
3. DB 스키마 정의
4. 백테스트 엔진 구현
5. 전략 3개 구현
6. 검증 리포트 생성
7. 실시간 모의투자 엔진 구현
8. Electron 앱 구현

## 11. 완료 정의

다음 조건을 모두 만족하면 MVP 완료로 본다.

- 주요 KRW 코인 5종에 대해 과거 캔들 적재 완료
- 전략 3개 이상 백테스트 가능
- 최근 1년 홀드아웃 성능 비교 가능
- 실시간 모의투자 실행 가능
- Electron 앱에서 모의투자 상태와 결과를 확인 가능
- 핵심 로그와 오류 정보를 저장 가능

## 12. MVP 이후 확장 항목

MVP 완료 후 다음 항목을 순차 검토한다.

- 실거래 주문 엔진
- 워크포워드 자동화
- 호가/체결 데이터 확장
- 전략 파라미터 탐색 자동화
- 전략 조합 포트폴리오
- 알림 시스템
- 운영 대시보드 고도화
