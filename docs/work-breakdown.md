# 업비트 자동매매 앱 작업 분해 문서

## 1. 문서 목적

이 문서는 MVP 기능 명세와 기술 설계 문서를 실제 구현 가능한 작업 단위로 분해한 문서다.

목표는 다음과 같다.

- 개발 순서를 고정한다.
- 병렬 작업 가능한 영역을 구분한다.
- 각 작업의 완료 기준을 명확히 한다.

## 2. 작업 원칙

- 의존성이 낮은 기반 작업부터 진행한다.
- 데이터 파이프라인을 UI보다 먼저 완성한다.
- 실거래 기능은 MVP 이후로 미룬다.
- 각 작업은 실행 가능 결과물을 남겨야 한다.

## 3. 전체 작업 흐름

구현 순서는 다음과 같다.

1. 저장소 초기화
2. 공통 설정 및 환경 구조 정리
3. 데이터베이스 스키마 정의
4. 데이터 수집기 구현
5. 백테스트 엔진 구현
6. 전략 구현
7. 검증 리포트 구현
8. 실시간 모의투자 엔진 구현
9. Electron 앱 구현
10. 통합 검증

## 4. 에픽 단위 작업

### Epic 1. 저장소 초기화

목표:

- 모노레포 골격 생성
- 서비스/앱/리서치 경계 확정

세부 작업:

- `apps/desktop` 디렉터리 생성
- `services/data-collector` 디렉터리 생성
- `services/paper-trader` 디렉터리 생성
- `research/backtester` 디렉터리 생성
- `research/strategies` 디렉터리 생성
- `packages/shared` 디렉터리 생성
- `infra/db` 디렉터리 생성
- 기본 `README.md` 작성
- 루트 `.gitignore` 작성
- 루트 환경변수 예시 파일 작성

완료 기준:

- 모든 상위 디렉터리와 기본 문서가 존재한다.
- 프로젝트 목적과 구조를 루트 README에서 이해할 수 있다.

### Epic 2. 공통 설정 및 개발 환경

목표:

- 앱과 서비스가 같은 기준으로 개발 가능하도록 설정

세부 작업:

- 루트 작업 규칙 정의
- Node 워크스페이스 초기화
- 공통 환경변수 키 정리
- 로컬 실행 스크립트 초안 작성

완료 기준:

- 개발자가 앱과 각 서비스의 실행 진입점을 알 수 있다.
- 환경변수 예시 파일만 보고 초기 실행 준비가 가능하다.

### Epic 3. 데이터베이스 스키마

목표:

- MVP에 필요한 핵심 테이블 정의

세부 작업:

- `markets` 스키마 정의
- `candles` 스키마 정의
- `collector_runs` 스키마 정의
- `backtest_runs` 스키마 정의
- `backtest_metrics` 스키마 정의
- `paper_sessions` 스키마 정의
- `paper_orders` 스키마 정의
- `paper_positions` 스키마 정의
- `system_logs` 스키마 정의
- 초기 migration 파일 작성

완료 기준:

- DB 초기화 스크립트 1회 실행으로 MVP 테이블이 생성된다.
- `candles` 중복 방지 제약조건이 포함된다.

### Epic 4. 데이터 수집기

목표:

- 업비트 과거 캔들 데이터를 안정적으로 적재

세부 작업:

- 업비트 REST 클라이언트 구현
- 마켓 목록 조회 기능 구현
- 캔들 단건 요청 래퍼 구현
- 페이지네이션 기반 수집 루프 구현
- 중복 제거 upsert 구현
- 수집 실행 로그 저장 구현
- 누락 구간 재수집 명령 구현
- CLI 엔트리포인트 작성

완료 기준:

- 지정 마켓/주기에 대해 과거 캔들 전체 적재가 가능하다.
- 재실행 시 중복 데이터가 쌓이지 않는다.

### Epic 5. 백테스트 엔진

목표:

- 동일 데이터셋에서 전략별 성능을 반복 가능하게 평가

세부 작업:

- 캔들 데이터 로더 구현
- 전략 인터페이스 정의
- 포지션 상태 모델 구현
- 주문 체결 모델 구현
- 수수료/슬리피지 모델 구현
- 에쿼티 커브 계산 구현
- 거래 로그 기록 구현
- 백테스트 실행 CLI 작성

완료 기준:

- 입력 데이터와 파라미터가 같으면 결과가 동일하다.
- 결과로 거래 로그와 핵심 지표가 생성된다.

### Epic 6. 전략 라이브러리

목표:

- 최소 3개 전략을 공통 인터페이스로 제공

세부 작업:

- 이동평균 교차 전략 구현
- 변동성 돌파 전략 구현
- RSI 평균회귀 전략 구현
- 전략 파라미터 스키마 정의
- 전략 레지스트리 구현

완료 기준:

- 전략 교체 시 백테스트 엔진 수정이 필요 없거나 최소화된다.

### Epic 7. 검증 리포트

목표:

- 최근 1년 홀드아웃 기준 비교 결과 제공

세부 작업:

- 학습/검증 구간 분리 로직 구현
- 전략별 반복 실행 로직 구현
- 지표 계산기 구현
- 결과 저장 로직 구현
- 비교표 출력 기능 구현

완료 기준:

- 전략별 학습 구간 성과와 검증 구간 성과를 분리해서 볼 수 있다.
- 비교 결과가 저장된다.

### Epic 8. 실시간 모의투자 엔진

목표:

- 선택 전략을 실시간 시세로 가상 운용

세부 작업:

- 업비트 WebSocket 클라이언트 구현
- 실시간 시세 상태 캐시 구현
- 세션 상태 머신 구현
- 전략 평가 루프 구현
- 가상 주문 생성 구현
- 가상 체결 구현
- 포지션/손익 갱신 구현
- 로그 및 상태 조회 API 구현
- 시작/중지 API 구현

완료 기준:

- 모의투자 세션을 시작하고 종료할 수 있다.
- 포지션과 손익이 실시간으로 갱신된다.

### Epic 9. Electron 앱

목표:

- 결과 조회와 모의투자 제어가 가능한 데스크톱 UI 제공

세부 작업:

- Electron 프로젝트 초기화
- 메인/프리로드/렌더러 분리
- 전략 목록 화면 구현
- 백테스트 결과 화면 구현
- 모의투자 대시보드 구현
- 연결 상태 표시 구현
- 로그 패널 구현

완료 기준:

- 앱에서 전략 조회와 모의투자 제어가 가능하다.
- 상태와 로그가 갱신된다.

### Epic 10. 통합 검증

목표:

- 각 모듈이 연결된 상태에서 MVP 동작 확인

세부 작업:

- 로컬 실행 순서 문서화
- 샘플 데이터로 초기 검증
- 데이터 적재 -> 백테스트 -> 모의투자 플로우 검증
- 주요 오류 상황 점검
- 기본 운영 체크리스트 작성

완료 기준:

- 문서만 따라도 전체 플로우를 재현 가능하다.

## 5. 병렬 가능 작업

다음 작업은 일부 병렬 진행이 가능하다.

- Epic 1과 Epic 2 일부
- Epic 3과 Epic 4 초기 준비
- Epic 5와 Epic 6
- Epic 8 API 설계와 Epic 9 UI 뼈대

단, 다음 선행관계는 유지해야 한다.

- DB 스키마 없이 안정적인 수집기 구현은 어렵다.
- 전략 인터페이스 없이 전략 구현을 확정하면 재작업 가능성이 크다.
- paper-trader API 없이 UI를 먼저 완성하면 연결부 재작업이 생긴다.

## 7. 위험 요소

### 기술 위험

- 업비트 rate limit 대응 미흡
- 시계열 데이터 누락 구간 발생
- 백테스트 로직에 미래 정보 누수 발생
- 모의투자와 백테스트 체결 규칙 불일치

### 프로젝트 위험

- UI를 먼저 만들다가 핵심 엔진이 밀리는 문제
- 전략 수만 늘리고 검증 품질이 낮아지는 문제
- 너무 이른 실거래 전환
