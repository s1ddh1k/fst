# 업비트 자동매매 Electron 앱 프로젝트 계획

## 1. 목표

이 프로젝트의 목표는 업비트 API를 이용해 다음 기능을 갖춘 자동매매 시스템을 구축하는 것이다.

- 몇 년치 시세 데이터를 수집하고 저장한다.
- 과거 데이터를 이용해 자동매매 전략을 학습하거나 튜닝한다.
- 최근 1년은 학습에 사용하지 않고 별도 검증 구간으로 유지한다.
- 여러 전략의 성능을 비교해 실전 투입 후보를 선정한다.
- Electron 앱에서 전략을 선택하고 실시간 모의투자를 수행한다.
- 모의투자 결과가 안정적일 때 실시간 자동매매로 전환한다.

핵심 원칙은 `백테스트 성과`보다 `실전 재현 가능성`을 우선하는 것이다.

## 2. 범위

### 포함 범위

- 업비트 Quotation API 및 WebSocket 기반 데이터 수집
- 과거 데이터 저장소 구축
- 전략 연구 및 백테스트 엔진 구축
- 홀드아웃 검증과 워크포워드 검증
- 실시간 모의투자 엔진 구축
- Electron 기반 데스크톱 앱 구축
- 이후 실거래 엔진으로 확장 가능한 구조 설계

### 제외 범위

- 초기에 모든 코인을 동시에 지원하는 기능
- 초기에 초단타 호가 기반 HFT 수준 전략
- 초기에 모바일 앱 개발
- 초기에 다중 거래소 연동

## 3. 성공 기준

다음 조건을 만족하면 1차 목표 달성으로 본다.

- 주요 KRW 마켓 코인에 대해 장기 시세 데이터 수집이 가능하다.
- 전략별 백테스트 결과를 반복 가능하게 재현할 수 있다.
- 학습 구간과 미학습 검증 구간이 명확히 분리된다.
- 모의투자 엔진이 실시간으로 주문, 포지션, 손익을 추적한다.
- Electron 앱에서 전략 선택, 성능 조회, 모의투자 실행이 가능하다.
- 수수료, 슬리피지, 손실 제한 등 실전 조건이 반영된다.

## 4. 현실적인 전제

자동매매 프로젝트는 기술적으로 구현 가능하지만, 다음 이유로 실제 수익 보장은 어렵다.

- 백테스트 과최적화 위험이 크다.
- 실제 체결 가격은 백테스트 가격보다 불리한 경우가 많다.
- 거래 수수료와 슬리피지가 누적되면 성과가 급격히 낮아질 수 있다.
- 급변장에서는 전략이 과거와 다르게 작동할 수 있다.

따라서 전략 선정 기준은 단순 수익률 1등이 아니라 다음 지표의 균형으로 잡아야 한다.

- 누적 수익률
- 최대 낙폭(Max Drawdown)
- Sharpe 또는 Sortino
- Profit Factor
- 거래당 기대값
- 검증 구간 일관성

## 5. 전체 아키텍처

시스템은 다음 6개 영역으로 분리한다.

### 5.1 데이터 수집기

- 업비트 REST API로 과거 캔들 데이터를 수집한다.
- 업비트 WebSocket으로 실시간 시세를 수신한다.
- API rate limit를 준수하면서 장기 적재를 수행한다.
- 수집 실패 시 재시도 및 누락 구간 복구 기능을 둔다.

### 5.2 데이터 저장소

- 캔들, 체결, 호가, 전략 결과, 주문 로그를 분리 저장한다.
- 연구용 분석 포맷과 운영용 저장 포맷을 구분한다.
- 원본 데이터와 파생 데이터를 분리한다.

### 5.3 연구 및 백테스트 엔진

- 전략 인터페이스를 공통 규격으로 정의한다.
- 수수료, 슬리피지, 최소 주문 금액, 포지션 사이즈를 반영한다.
- 파라미터 탐색과 결과 기록을 자동화한다.

### 5.4 검증 엔진

- 최근 1년 홀드아웃 검증을 수행한다.
- 추가로 워크포워드 검증을 수행한다.
- 여러 기간에서 반복 검증하여 과최적화를 줄인다.

### 5.5 실시간 모의투자 엔진

- WebSocket 기반 실시간 가격 반영
- 가상 주문 및 체결 처리
- 포지션, 잔고, 미실현 손익 추적
- 리스크 제한 적용

### 5.6 Electron 앱

- 전략 목록 및 성능 비교 화면
- 백테스트 결과 조회
- 실시간 모의투자 실행 및 중지
- 포지션, 손익, 주문 로그, 시스템 상태 표시

## 6. 권장 기술 스택

초기 단계에서는 스택을 최대한 단일화하는 편이 구현과 운영에 유리하다.

- 앱 및 서비스: Node.js + TypeScript
- 데스크톱 앱: Electron + React + TypeScript
- DB: PostgreSQL + TimescaleDB 또는 PostgreSQL 단독
- 분석용 파일 포맷: CSV 또는 Parquet
- 차트: TradingView Lightweight Charts
- 스케줄링: cron 또는 Node 기반 작업 러너

권장 방향은 다음과 같다.

- 데이터 수집, 백테스트, 모의투자를 모두 TypeScript로 통일
- 앱 UI는 Electron + React
- 앱과 매매 엔진은 프로세스를 분리

## 7. 데이터 전략

### 7.1 우선 수집 대상

초기에는 다음 범위만 우선 지원한다.

- KRW-BTC
- KRW-ETH
- KRW-XRP
- KRW-SOL
- KRW-DOGE

이후 점진적으로 종목을 확장한다.

### 7.2 우선 수집 주기

초기에는 다음 데이터를 우선 수집한다.

- 1분봉
- 5분봉
- 1시간봉
- 일봉

체결 데이터와 호가 데이터는 2차 단계에서 추가한다.

### 7.3 저장 원칙

- 원본 응답은 가능한 한 손실 없이 저장
- 분석용 정규화 테이블 별도 생성
- 타임존 일관성 유지
- 데이터 누락 구간 기록

## 8. 전략 개발 방향

초기 전략은 설명 가능하고 해석 가능한 전략 위주로 시작한다.

### 8.1 1차 전략 후보

- 이동평균 교차
- 변동성 돌파
- RSI 기반 평균회귀
- Bollinger Band 기반 평균회귀
- 추세 필터 + 손절/익절 조합

### 8.2 전략 선택 원칙

- 단순한 전략부터 기준선을 만든다.
- 지나치게 많은 파라미터를 갖는 전략은 지양한다.
- 수익률만이 아니라 낙폭과 거래 빈도를 함께 본다.

### 8.3 머신러닝 적용 원칙

머신러닝은 초기에 필수가 아니다. 다음 순서가 적절하다.

1. 규칙 기반 전략으로 기준 성능 확보
2. 피처 엔지니어링 체계 확립
3. 가벼운 분류/회귀 모델 도입
4. 실전 재현성 확인 후 고급 모델 검토

초기부터 딥러닝에 바로 진입하는 것은 비효율적일 가능성이 높다.

## 9. 학습 및 검증 계획

### 9.1 기본 분할

- 전체 데이터 중 최근 1년은 홀드아웃 검증 구간으로 분리
- 나머지 기간은 학습 및 파라미터 탐색에 사용

예시:

- 2021-01-01 ~ 2024-12-31: 학습
- 2025-01-01 ~ 2025-12-31: 미학습 검증

### 9.2 추가 검증

홀드아웃 1회만으로는 부족하므로 워크포워드 검증을 병행한다.

예시:

- 2021-2023 학습, 2024 검증
- 2022-2024 학습, 2025 검증

### 9.3 평가 기준

전략 비교 시 다음 항목을 모두 기록한다.

- 총 수익률
- 연환산 수익률
- 최대 낙폭
- Sharpe/Sortino
- 승률
- Profit Factor
- 거래 횟수
- 평균 보유 시간
- 검증 구간 성과 저하율

## 10. 모의투자 계획

실시간 모의투자는 실거래 전 필수 단계로 둔다.

### 목표

- 실시간 데이터 수신 안정성 검증
- 전략 신호 생성 지연 확인
- 가상 체결 로직 검증
- 손익 계산 및 포지션 추적 검증
- 리스크 제어 검증

### 필수 조건

- 실거래와 동일한 수수료 반영
- 보수적인 슬리피지 반영
- 주문 실패/지연 상황 시뮬레이션 가능
- 실시간 로그 저장

## 11. 실거래 전환 계획

실거래는 다음 조건을 만족할 때만 허용한다.

- 모의투자 결과가 충분한 기간 동안 안정적이다.
- 검증 구간 성과가 백테스트 대비 과도하게 붕괴하지 않는다.
- 리스크 제한 기능이 검증되었다.
- 장애 상황에서 자동 정지 기능이 동작한다.

실거래 초기 원칙은 다음과 같다.

- 소액으로 시작
- 전략 1개만 투입
- 거래 대상 종목 제한
- 일 최대 손실 한도 설정
- 수동 정지 버튼 상시 제공

## 12. 보안 및 리스크 관리

### 보안 원칙

- 업비트 API Secret은 렌더러 프로세스에 노출하지 않는다.
- 민감 정보는 OS 키체인 또는 암호화 저장소를 사용한다.
- 주문 로직은 별도 프로세스 또는 백엔드 레이어에서 처리한다.

### 리스크 제어

- 일간 최대 손실 한도
- 종목당 최대 비중
- 최대 동시 포지션 수
- 연속 손실 시 자동 중지
- WebSocket 단절 시 거래 중지
- 잔고 이상 감지 시 거래 중지

- 홀드아웃 검증 자동화

### Phase 6. 실시간 모의투자 구축

## 14. 일정 제안

12주 기준 MVP 일정 예시는 다음과 같다.

- 1-2주: 저장소 초기화, 아키텍처 확정, DB 설계
- 3-4주: 업비트 데이터 수집기 구현
- 5-6주: 백테스트 엔진 및 전략 1차 구현
- 7-8주: 검증 자동화 및 성능 리포트
- 9-10주: 실시간 모의투자 엔진 구현
- 11주: Electron 앱 대시보드 구현
- 12주: 통합 테스트 및 실거래 베타 준비

## 15. MVP 정의

초기 MVP는 다음 기능까지를 목표로 한다.

- KRW 주요 코인 과거 데이터 수집
- 전략 3~5개 백테스트 가능
- 최근 1년 홀드아웃 검증 가능
- 실시간 모의투자 1개 전략 실행 가능
- Electron 앱에서 결과 조회 및 모의투자 제어 가능

실거래는 MVP 완료 이후 2차 목표로 둔다.

## 18. 결론

이 프로젝트는 충분히 구현 가능하다. 다만 성공 가능성을 높이려면 `앱 개발`보다 먼저 `데이터 수집`, `검증 가능한 백테스트`, `실시간 모의투자`, `리스크 통제`를 우선 완성해야 한다.

가장 중요한 목표는 “가장 높은 수익률 전략”이 아니라 “미학습 구간과 실시간 구간에서도 무너지지 않는 전략”을 찾는 것이다.
