# AI Native Auto Research System Requirements

버전: v0.1  
대상 저장소: `fst`  
작성일: 2026-03-17  
목적: 사람이 매번 아이디어를 주고 튜닝하는 방식이 아니라, **LLM이 스스로 아이디어 생성, 리서치, 코드 수정, 데이터 준비, 백테스트, 평가, 폐기/승격 판단**을 반복하는 자동 연구 애플리케이션의 요구사항을 정의한다.

---

## 1. 한 줄 목표

이 시스템의 목표는 다음이다.

- Upbit KRW 현물 롱 전용 시스템에 대해
- LLM이 전략 아이디어를 스스로 만들고
- 필요하면 데이터를 더 수집하고, 파생 피처를 캐시하고
- 전략 코드와 연구 코드의 버그를 직접 수정하고
- 백테스트와 리포트를 다시 실행하고
- 성과가 약하면 조정하거나 폐기하고
- 충분히 납득 가능한 후보만 승격시키는
- **자동 연구 루프**를 만드는 것

즉, 단순한 “파라미터 스윕 툴”이 아니라 **AI 연구원 + 실험 실행기 + 코드 에이전트**가 결합된 구조가 필요하다.

---

## 2. 왜 필요한가

현재 연구 흐름의 문제는 다음과 같다.

- 사람이 전략 아이디어를 직접 넣어야 한다.
- 사람이 중간중간 결과를 보고 다음 액션을 결정해야 한다.
- 전략이 안 좋을 때도 폐기 판단이 늦다.
- 버그 수정, 데이터 준비, 리포트 확인이 분리돼 있다.
- 전략 카탈로그가 고정돼 있어 LLM이 새로운 실험 공간을 넓히기 어렵다.

원하는 구조는 다음과 다르다.

- 사람이 “이 전략을 이렇게 바꿔봐”라고 계속 말하는 연구 방식

원하는 구조는 다음과 같다.

- 사람이 목표와 제약만 준다.
- LLM이 실험 가설을 만들고 실행한다.
- 필요하면 코드를 바꾸고 다시 검증한다.
- 충분히 괜찮은 결과가 나올 때까지 반복한다.

---

## 3. 제품 정의

이 시스템은 다음 세 가지를 동시에 수행하는 애플리케이션이어야 한다.

1. **Research Planner**
   - 아이디어 생성
   - 관련 전략 철학 조사
   - 다음 실험 제안
   - 유지/조정/폐기 판단

2. **Execution Worker**
   - 데이터 준비
   - 파생 피처 캐시 준비
   - 백테스트 실행
   - 리포트 수집

3. **Code Agent**
   - 전략 코드 수정
   - 연구용 오케스트레이터/리포트 코드 수정
   - 버그 발견 시 수정
   - 전략 카탈로그 확장

---

## 4. 핵심 요구사항

### 4.1 LLM 주도 아이디어 생성

필수 요구사항:

- 사람이 전략 아이디어를 직접 주지 않아도 돼야 한다.
- LLM이 스스로 다음 후보 전략 또는 후보 변형을 제안해야 한다.
- 제안은 단순 파라미터 미세조정에만 머물면 안 된다.
- LLM은 다음을 함께 제안할 수 있어야 한다.
  - 새로운 전략 패밀리
  - 기존 전략의 구조적 변경
  - 특정 전략의 폐기
  - 데이터 추가 수집 필요성
  - 파생 피처 캐시 필요성
  - 평가 기준 조정 필요성

권장:

- 각 제안에는 짧은 thesis와 invalidation 조건이 있어야 한다.
- 각 제안은 “왜 이 전략을 지금 탐색해야 하는가”를 짧게 설명해야 한다.

### 4.2 LLM 리서치 동반

필수 요구사항:

- 아이디어 생성은 단순 랜덤 변형이 아니라 리서치가 동반되어야 한다.
- 리서치는 다음 중 일부를 포함할 수 있어야 한다.
  - 기존 저장소 문맥 분석
  - 이전 백테스트 결과 해석
  - 시장 구조 관점 reasoning
  - 전략 철학 비교
  - 필요 시 외부 정보/문헌/웹 조사

주의:

- 리서치 결과는 실행 가능한 실험 제안으로 이어져야 한다.
- “설명만 하는 답변”으로 끝나면 안 된다.

### 4.3 코드 수정 가능 구조

필수 요구사항:

- LLM은 단순히 파라미터만 바꾸는 것이 아니라 **코드를 수정할 수 있어야 한다.**
- 다음 종류의 코드 변경을 할 수 있어야 한다.
  - 전략 코드 수정
  - 전략 카탈로그 추가
  - 리포트/평가 코드 수정
  - 데이터 준비 코드 수정
  - 백테스트 엔진 버그 수정
  - 연구 오케스트레이터 자체 개선

필수 조건:

- 코드 변경 후 자동 테스트 또는 검증 루프가 따라야 한다.
- 변경은 artifact와 함께 남아야 한다.
- 실패한 변경은 폐기 가능해야 한다.

### 4.4 전략 카탈로그의 동적 확장

필수 요구사항:

- “실행 가능한 전략 카탈로그”가 고정이면 안 된다.
- LLM은 새 전략 패밀리를 추가하거나, 기존 전략을 더 조사하고, 실험 가능한 카탈로그를 확장할 수 있어야 한다.

이를 위해 카탈로그는 두 층으로 나뉘어야 한다.

1. **Stable Catalog**
   - 이미 구현되어 즉시 실행 가능한 전략 패밀리

2. **Proposed Catalog**
   - LLM이 새로 제안한 전략 패밀리
   - 아직 코드 생성/수정이 필요함
   - 검토/구현/실험 후보 상태를 가짐

필수 기능:

- 카탈로그 항목 상태 추적
  - `proposed`
  - `implemented`
  - `validated`
  - `discarded`
- 각 패밀리의
  - thesis
  - 파라미터 스키마
  - 필요 데이터
  - 평가 기준
  - 구현 상태
  를 기록해야 한다.

### 4.5 자동 데이터 준비

필수 요구사항:

- 연구 과정에서 필요하면 시스템이 직접 데이터 준비를 요청하고 실행할 수 있어야 한다.

지원 대상:

- 분봉/시간봉 원천 데이터 수집
- 누락 구간 backfill
- universe refresh
- 파생 피처 캐시 생성
- feature cache 재계산

예:

- “15m 데이터가 부족하면 수집”
- “1h breadth/relative strength cache가 반복 계산 병목이면 저장”
- “최근 데이터 freshness가 부족하면 sync latest”

필수 조건:

- 데이터 준비 액션도 실험 artifact에 남아야 한다.
- 어떤 실험이 어떤 데이터 상태에서 수행됐는지 추적 가능해야 한다.

### 4.6 자동 버그 수정 루프

필수 요구사항:

- 연구 도중 버그가 발견되면 LLM이 이를 단순 보고하는 것으로 끝내면 안 된다.
- 다음 순서가 자동화돼야 한다.

1. 실패 원인 추정
2. 관련 코드/테스트 탐색
3. 수정
4. 테스트/재실행
5. 결과 기록

대상 버그 예:

- same-bar fill 판정 오류
- 거래 집계 오류
- 지표 계산 오류
- 전략 카탈로그 파라미터 매핑 오류
- 리포트 집계 오류
- 피처 캐시 정합성 문제

### 4.7 평가와 폐기 자동화

필수 요구사항:

- 백테스트 결과가 약하면 LLM이 다음 중 하나를 결정해야 한다.
  - 유지
  - 재설계
  - 추가 실험
  - 폐기

폐기 판단 기준 예:

- 거래 수가 너무 적음
- 비용 반영 후 edge 없음
- 특정 종목 하나에만 편중
- 특정 기간에만 성과 발생
- drawdown/turnover가 과도함
- bootstrap/random benchmark에서 약함

즉, “어떻게든 살려보자”가 아니라 **냉정하게 버릴 수 있는 구조**가 필요하다.

### 4.8 Artifact 중심 운영

필수 요구사항:

- 모든 자동 연구 실행은 artifact를 남겨야 한다.

최소 artifact:

- 실행 시각
- 연구 목표
- LLM proposal
- 데이터 준비 액션
- 코드 수정 요약
- 실행된 백테스트 조건
- 결과 메트릭
- review verdict
- 다음 액션

권장 artifact:

- iteration별 markdown report
- structured JSON report
- 변경된 파일 목록
- 후보 전략 상태 변화

---

## 5. 운영 제약

다음 제약은 유지해야 한다.

- TypeScript 유지
- Upbit KRW 현물
- 롱 전용
- lookahead 금지
- same-bar fill 금지
- point-in-time universe 유지
- 거래소 제약은 전략 코드 밖에 둔다

추가 운영 제약:

- LLM이 창의적으로 전략을 제안해도 위 제약은 깨면 안 된다.
- 안전하지 않은 무제한 코드 생성/실행은 금지한다.
- 코드 수정은 검증 단계를 반드시 거쳐야 한다.

## 6. 비목표

이번 시스템의 1차 목표는 아니다.

- 완전 자율 라이브 운용
- 무제한 외부 웹 crawling
- orderbook microstructure 실시간 학습 자동화
- 모든 전략을 코드 생성만으로 바로 생산 등급화

즉, 1차 목표는 **자율 연구와 자율 검증**이다.

## 8. 완료 기준

다음 조건을 만족해야 “초기 버전 완성”으로 본다.

- LLM이 실행 가능한 후보 전략을 스스로 제안할 수 있다.
- 후보 전략을 자동으로 백테스트할 수 있다.
- 결과를 바탕으로 LLM이 keep/refine/discard를 결정할 수 있다.
- 필요 시 데이터 수집/feature cache 액션을 실행할 수 있다.
- 버그 수정 가능한 코드 에이전트 경로가 구조적으로 준비된다.
- 전략 카탈로그를 LLM이 확장할 수 있는 모델이 존재한다.
- 모든 과정이 artifact로 남는다.

---

## 9. 핵심 설계 원칙

- 휴리스틱 자동화가 아니라 **LLM 주도 연구 루프**
- 수동 티키타카가 아니라 **자율 제안과 자율 평가**
- 파라미터 스윕만이 아니라 **구조적 전략 생성과 코드 수정**
- 설명 중심이 아니라 **실행과 결과 중심**
- 사람은 목표와 제약을 주고, 시스템은 실험과 검증을 수행
