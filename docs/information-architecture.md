# fst Desktop Information Architecture

## 목적

이 문서는 `apps/desktop`의 사용자 정보 구조를 정리하고, 현재 운영 대시보드를 과업 중심 데스크톱 앱으로 재구성하기 위한 기준을 정의한다.

핵심 목표는 아래와 같다.

- 사용자가 현재 시스템 상태를 빠르게 파악할 수 있다.
- 전략 비교와 세션 운영을 분리해 예측 가능한 탐색 구조를 만든다.
- 운영 정보는 필요할 때만 접근하는 보조 흐름으로 격리한다.

## 사용자 과업

데스크톱 앱의 주요 과업은 다음 네 가지다.

1. 현재 상태를 한눈에 확인한다.
2. 추천 전략과 검증 결과를 비교한다.
3. 세션을 생성하고 실행 상태를 모니터링한다.
4. 장애나 운영 이슈가 있을 때 런타임을 점검한다.

## 상위 정보 구조

앱은 다음 네 개의 상위 허브로 구성한다.

### 1. 개요

역할:

- 현재 시스템과 추천/세션 상태를 5초 안에 파악하는 허브

포함 정보:

- 시스템 상태 배너
- 추천 전략 수
- 활성 세션 수
- 최신 collector / paper 로그
- 추천 결과 세트 요약
- 최근 세션 요약

### 2. 전략

역할:

- 어떤 전략을 세션으로 올릴지 결정하는 허브

포함 정보:

- 추천 전략 목록
- 추천 결과 세트 상세
- 검증 기간과 규칙
- 비교용 핵심 성능 지표
- 세션 생성 액션

### 3. 세션

역할:

- 이미 생성한 세션을 실행하고 상태를 추적하는 허브

포함 정보:

- 세션 목록
- 세션 상세
- 포지션 / 손익
- 최근 주문
- 타임라인
- 실행 액션

### 4. 운영

역할:

- 인프라 및 로컬 런타임 점검 허브

포함 정보:

- paper-trader 상태
- API 상태
- tmux 워크스페이스
- 운영 명령
- 로그 경로

## Sitemap

```text
fst desktop
├─ 개요
│  ├─ 시스템 상태
│  ├─ 핵심 지표
│  ├─ 추천 결과 세트 요약
│  └─ 최근 세션 요약
├─ 전략
│  ├─ 추천 전략 목록
│  ├─ 비교 지표
│  ├─ 추천 결과 세트
│  └─ 세션 생성
├─ 세션
│  ├─ 세션 목록
│  ├─ 세션 상세
│  ├─ 포지션 / 손익
│  ├─ 주문 / 타임라인
│  └─ 실행
└─ 운영
   ├─ 시스템 상태
   ├─ tmux 워크스페이스
   ├─ 운영 명령
   └─ 로그 정보
```

## 내비게이션 원칙

- 1차 내비게이션은 `개요 / 전략 / 세션 / 운영` 네 개로 제한한다.
- `마켓 / 주기 / 시작 잔고`는 전역 탐색이 아니라 `실행 설정`으로 분류한다.
- `tmux`, 런북, 로그는 `운영`에 배치한다.
- 전략 선택과 세션 실행은 다른 허브로 분리하되, 전략 허브에서 세션 생성으로 이어지는 직접 액션은 유지한다.

## 라벨 원칙

- 내부 구현 용어보다 사용자 과업 용어를 우선한다.
- 짧고 예측 가능한 명사를 쓴다.
- 동일한 객체는 앱 전역에서 같은 이름을 사용한다.

권장 라벨:

- `트리거` -> `실행 설정`
- `운영 보드` -> `운영 상태`
- `세션 포커스` -> `세션 상세`
- `추천 스냅샷` -> `추천 결과 세트`

## 구현 원칙

- 1차 구현은 라우터 없이 로컬 상태 기반 허브 전환으로 시작한다.
- 기존 데이터 훅과 패널은 재사용하고, 상위 구조만 재배치한다.
- 이후 라우팅이 필요해질 때 동일한 허브 구조를 그대로 URL 체계로 확장한다.
